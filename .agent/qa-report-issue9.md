## Code Review

### Correctness

**Period timing** PASS
`changePeriod()` calls `generateScoreReport()` before incrementing `this.gameState.period`. The score report correctly names the outgoing period. `_autoPlayPeriodScore()` receives the text synchronously after `generateScoreReport()` sets `sequenceBuilder.generatedText`. No timing issues.

**Progressive degradation** PASS
`_autoPlayPeriodScore()` guards:
1. Empty text → return
2. No API key (`storage.getApiKey()`) → return silently
3. No voice ID (`storage.getVoiceId()`) → return silently
4. TTS exception → `console.warn` + fall through (text remains in audio bar for manual play)

All paths covered. Operator can always press PLAY manually as fallback.

**State consistency** PASS
`autoAnnouncePeriodEnd` is persisted alongside `homeScore`, `awayScore`, and `period` in `gameState` → `localStorage`. Migration guard correctly defaults new field to `true` for existing game states. `populateSettings()` and `saveSettings()` correctly read/write the checkbox.

**TTS instance integrity** PASS
`this.tts` (`ElevenLabsTTS`) is initialized at app startup with key/voice from storage. `saveSettings()` calls `this.tts.setApiKey()` and `this.tts.setVoice()` when settings change. The guards in `_autoPlayPeriodScore()` that check `storage.getApiKey()` / `storage.getVoiceId()` are belt-and-suspenders: the TTS instance will throw (and be caught) if its internal state is somehow stale.

**Audio channel routing** PASS
`this.audioManager.play(audio)` routes to the right/PA channel (pan=+1) at full volume — same path as all manual PLAY presses. Correct behavior.

## Findings

### L1 — Async race: double TTS call if operator presses PLAY during ElevenLabs latency window (LOW)

**File:** `js/app.js` — `_autoPlayPeriodScore()` / `js/audio-manager.js` — `playWithPan()`

`_autoPlayPeriodScore()` is fire-and-forget async. During the ~1-2s ElevenLabs round-trip, if the operator presses PLAY in the audio bar, a second TTS call starts. Both calls end up calling `audioManager.play()` → `playWithPan()`. Each invokes `this.stop()` at its own entry, but since both are concurrent async calls (awaiting `arrayBuffer()` and `decodeAudioData()`), neither reliably stops the other's decoded source before it plays.

**Consequence:** Brief audio overlap on the PA channel — two slightly desynchronized reads of the same score text. No data loss, no crash.

**Assessment:** LOW severity. Race window is narrow (~1-2s). Scenario requires operator pressing PLAY within 1-2 seconds of advancing a period. Text is always visible in the audio bar as fallback. Does not block ship.

**Fix if needed:** An `_autoPlayInFlight` boolean guard in `_autoPlayPeriodScore()` would prevent re-entrant calls cleanly.
